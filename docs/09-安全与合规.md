# 09 - 安全与合规设计

## 9.1 安全设计哲学

> **安全不是一个独立模块，而是融入 Agent 每一层的设计思维。**

MediMate 的安全机制**完全通过 System Prompt 实现**，不依赖关键词列表或正则匹配。

原因：
- LLM 能理解语义和上下文，而正则只能匹配字面量
- "吃了一瓶药想自杀" vs "研究自杀基因的论文" —— LLM 能区分，正则不能
- "我胸口有点闷" vs "药品说明书上写的胸闷" —— LLM 能判断紧急程度，正则会误报

## 9.2 System Prompt 中的安全规则

安全规则在 System Prompt 中被标记为**最高优先级**，LLM 会优先执行：

### 紧急情况处理

```
当用户描述以下任何情况时，立即停止回答用药问题，转而劝导就医：
- 严重的身体症状（胸痛、呼吸困难、大量出血、意识模糊、严重过敏反应等）
- 药物过量或误服
- 自杀/自残意念

紧急情况的回复必须包含：
- 明确建议立即就医
- 急救电话 120
- "请不要依赖在线工具处理紧急情况"
```

### 行为禁区

```
- 绝对不编造药物信息
- 绝对不提供处方建议
- 绝对不鼓励用户自行调整用药方案
- 绝对不说"你应该停药/换药"
```

### 为什么这样做就够了？

| 疑问 | 回应 |
|------|------|
| LLM 会不会遗漏紧急情况？ | 现代 LLM（如 DeepSeek）经过对齐训练，对安全相关输入天然敏感。System Prompt 的高优先级标注进一步强化了这一点。 |
| 没有确定性兜底会不会不安全？ | 现已加入正则硬兜底层作为前置分类器，与 LLM 语义判断形成双重保险，确保零容忍高危关键词 100% 拦截。 |
| 用户故意绕过呢？ | 这是所有 LLM 产品的共性问题。对于 MediMate 这个场景，用户故意绕过安全限制去获取"我应该停药"的回复，动机极弱。 |

## 9.3 医疗安全设计原则

| 原则 | 如何实现 |
|------|---------|
| **不替代医生** | System Prompt 明确定义"你是信息助手，不是医生" |
| **不做诊断** | 禁止说"你得了XX"，只允许展示工具返回的药物信息 |
| **不调整处方** | 禁止说"你应该停/换/加药"，只允许"建议咨询医生" |
| **信息有据可查** | 只使用工具返回的数据，禁止 LLM 自行补充药物知识 |
| **紧急优先** | 识别到紧急症状时终止药物对话，引导就医 |

## 9.4 免责声明体系

| 层级 | 位置 | 内容 | 实现方式 |
|------|------|------|---------|
| **全局免责** | 页面底部固定 | "本工具仅供健康参考，不构成医疗建议。用药请遵医嘱。" | 前端固定文本 |
| **首次确认** | 进入时弹出 | 完整免责声明，用户需点击"我已了解" | 前端模态框，确认状态存 LocalStorage |
| **回复级免责** | Agent 每轮回复末尾 | "⚕️ 以上信息仅供参考，请遵医嘱" | System Prompt 指示 LLM 附加 |
| **数据级免责** | FDA 数据展示时 | "报告数据不代表因果关系" | System Prompt 指示 LLM 说明 |

### 首次免责声明内容

```
⚕️ 使用须知

MediMate 是一款 AI 用药信息查询工具。

在使用前，请了解：
1. 本工具提供的信息仅供参考，不构成任何医疗建议、诊断或处方
2. 药物使用请严格遵循医生处方和药品说明书
3. 如出现任何不适症状，请立即就医
4. 本工具不收集任何个人健康数据

[我已了解，开始使用]
```

## 9.5 数据隐私设计

| 措施 | 说明 |
|------|------|
| **API Key 在后端** | DeepSeek API Key 存在后端 `.env` 中，不暴露给前端 |
| **对话不持久化** | 对话历史仅在会话期间存在于后端内存中，关闭后清除 |
| **用药清单可清除** | 用户可一键清空 PostgreSQL 中的用药记录 |
| **无账号体系** | 不收集姓名、手机号等任何个人身份信息 |
| **匿名会话** | 使用随机 UUID 作为 session_id，不与用户身份关联 |
| **FDA 查询匿名** | 调用 OpenFDA API 仅传药物名称，不传任何用户信息 |
| **LLM 数据传输** | 对话内容发送至 DeepSeek API，遵守其数据使用协议 |

> **注意**：使用第三方 LLM API 意味着对话内容会经过第三方服务器。在产品正式上线前，需评估 DeepSeek 的数据使用政策，并在免责声明中告知用户。

## 9.6 合规定位

| 维度 | MediMate 的定位 |
|------|----------------|
| 是否属于医疗器械？ | ❌ 否 — 仅提供药物公开信息查询，不做诊断、不推荐治疗方案 |
| 是否涉及个人健康数据？ | ⚠️ 用药清单可视为轻度健康数据 — 存储于 Supabase PostgreSQL（云端托管），通过匿名 UUID 关联，不绑定个人身份 |
| 数据来源合规性 | ✅ 合规 — 药物知识来自公开药品说明书，不良反应数据来自 OpenFDA 公开 API |
| LLM API 合规性 | ⚠️ 需确认 DeepSeek 数据使用政策，确保对话内容不被用于训练 |

## 9.7 安全迭代路径

如果后续需要加强安全性，可以按以下路径迭代：

| 阶段 | 措施 | 复杂度 |
|------|------|--------|
| 当前版本 | System Prompt 安全规则 + 正则硬兜底 | 低代码 |
| 下一版本 | 引入专门的医疗安全审核模型（如微调的 BERT 分类器） | 中 |
| 远期 | 对接专业医疗知识图谱做事实核查 | 高 |

当前阶段的双层安全机制（LLM 语义判断 + 正则硬兜底）已能覆盖医疗场景的安全需求。
