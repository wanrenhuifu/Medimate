# 12 - 风险分析

## 12.1 风险矩阵

| 风险 | 可能性 | 影响 | 风险等级 | 应对策略 |
|------|--------|------|---------|---------|
| **医疗信息不准确** | 中 | 高 | 🔴 高 | 数据来源标注 + 免责声明 + 限定覆盖范围 |
| **用户过度依赖** | 中 | 高 | 🔴 高 | 持续提示"仅供参考"，紧急情况强制引导就医 |
| **OpenFDA API 不可用** | 低 | 中 | 🟡 中 | 降级到内置模拟数据，标注"参考数据" |
| **药物名称识别失败** | 中 | 低 | 🟡 中 | 模糊匹配 + 同义词映射 + 追问确认 |
| **合规风险** | 低 | 高 | 🟡 中 | 纯信息展示，不做诊断/处方，免责声明完善 |
| **演示失败** | 低 | 高 | 🟡 中 | 全栈架构，DeepSeek/OpenFDA API 不可用时有降级策略 |
| **数据覆盖不足** | 中 | 低 | 🟢 低 | 诚实告知未收录，引导查阅说明书 |
| **用户隐私泄露** | 极低 | 高 | 🟢 低 | 匿名会话 UUID，数据不关联用户身份，无账号体系 |

## 12.2 关键风险详细分析

### 风险 1：医疗信息不准确

| 维度 | 分析 |
|------|------|
| **触发条件** | 内置知识库数据过时或有误；用户查询的场景超出覆盖范围 |
| **影响** | 用户基于错误信息做出用药决策，可能造成健康风险 |
| **缓解措施** | ① 数据仅来源于权威渠道（药品说明书、FDA）<br/>② 每条回复标注来源<br/>③ 全程免责声明<br/>④ 不做诊断和处方推荐 |
| **残余风险** | 用户忽视免责声明而过度信赖 → 通过 UI 设计加强免责声明可见性 |

### 风险 2：用户过度依赖

| 维度 | 分析 |
|------|------|
| **触发条件** | 用户将 MediMate 当作医生替代品，不再就医 |
| **影响** | 延误治疗，特别是严重疾病 |
| **缓解措施** | ① 紧急症状强制就医引导<br/>② 不回答"我该不该去看医生"这类问题（一律建议就医）<br/>③ 定期提醒"如有持续不适，建议就医" |

### 风险 3：面试演示失败

| 维度 | 分析 |
|------|------|
| **触发条件** | 面试现场网络不好导致 FDA API 调用失败；浏览器兼容性问题 |
| **影响** | 无法展示核心功能，影响面试评价 |
| **缓解措施** | ① API 降级机制（OpenFDA 不可用时优雅降级）<br/>② DeepSeek API 不可用时返回明确错误提示<br/>③ 提前在常用浏览器测试<br/>④ 准备离线演示版本（内置模拟数据兜底） |

## 12.3 伦理考量

| 考量 | MediMate 的做法 | 原理 |
|------|----------------|------|
| **信息准确性** | 仅使用权威来源数据，不使用 AI 生成的医学结论 | 医疗信息错误的代价远高于其他领域 |
| **认知偏差** | 避免锚定效应（不强调某个副作用），提供全面信息 | 不让用户因 Agent 的展示方式产生不合理的恐惧 |
| **数字鸿沟** | 界面简洁易用，支持大字体，降低老年用户门槛 | 最需要用药帮助的往往是老年人 |
| **过度医疗化** | 不鼓励用户"没事也要查"，而是在有需要时提供帮助 | 避免加剧健康焦虑 |
| **透明性** | Agent 明确告知自己"不是医生"，展示能力边界 | 让用户在知情的基础上使用 |

## 12.4 竞争风险

| 风险 | 可能性 | 应对 |
|------|--------|------|
| 大厂推出类似产品 | 高 | MediMate 定位为简历项目，不追求商业竞争 |
| 现有竞品增加 Agent 能力 | 中 | 持续关注市场动态，保持产品差异化 |
| LLM 能力提升使工具调用不再新鲜 | 中 | 关注 Agent 设计模式演进，保持学习 |
